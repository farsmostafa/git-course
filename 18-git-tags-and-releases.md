# 📝 الإصدارات والعلامات (Git Tags & Releases) - الدرس الـ 18

في الدرس ده، اتعلمنا إزاي نقسم المشروع بتاعنا لـ "أجيال" أو "إصدارات" (Versions). زي ما `Bootstrap` عنده إصدار ثالث ورابع وخامس، إنت كمان تقدر تعمل للمشروع بتاعك علامة مميزة (`Tag`) توثق بيها حالة المشروع في نقطة زمنية معينة كإصدار مستقر.

## 1. ما هو الـ `Tag`؟

- **المفهوم:** هو مجرد "علامة مرجعية" ثابتة بتتحط على `Commit` معين.
- **الهدف:** بيُستخدم غالباً لتحديد الإصدارات (مثلاً: `v1.0`, `v2.0-beta`)، بحيث لو المشروع اتطور وظهر فيه مشاكل بعدين، يقدر أي حد يرجع للإصدار `v1.0` القديم ويحمله كأنه كان واقف في الزمن عند اللحظة دي.

## 2. أنواع الـ `Tags` في `Git`

عندنا نوعين أساسيين:

### النوع الأول: `Lightweight Tag` (الخفيف)

- ده نوع بسيط جداً، بيعمل علامة على المكان اللي إنت فيه دلوقتي من غير ما يطلب منك رسالة وصف.
- **عيبه:** بياخد نفس الرسالة المكتوبة في آخر `Commit` إنت عملته ويسجلها لنفسه كرسالة للإصدار.

**القاعدة الأساسية (Syntax):**

    git tag <tag_name>

**المثال العملي (Example):**

    git tag v1.0

### النوع الثاني: `Annotated Tag` (المفصل/المشروح)

- ده النوع الاحترافي والأكثر استخداماً في الشركات، لأنه بيسمحلك تكتب رسالة أو وصف خاص بالإصدار ده بغض النظر عن رسالة الـ `Commit`. بنستخدم معاه حرف `-a` لتعريفه، وحرف `-m` لكتابة الرسالة.

**القاعدة الأساسية (Syntax):**

    git tag -a <tag_name> -m "<Description message>"

**المثال العملي (Example):**

    git tag -a v2.0 -m "Our project second version with new features"

## 3. عرض الإصدارات الحالية

عشان تشوف كل الـ `Tags` اللي إنت عملتها في مشروعك لحد دلوقتي:

**القاعدة الأساسية والمثال العملي:**

    git tag

_(سيظهر لك قائمة مثل: `v1.0`، `v2.0`)._

## 4. رفع الإصدارات إلى `GitHub`

**معلومة هامة:** أمر `git push` العادي اللي بنرفع بيه الأكواد **لا يرفع** الـ `Tags`. لازم ترفع كل `Tag` بنفسك بشكل صريح.

**القاعدة الأساسية (Syntax):**

    git push <remote_name> <tag_name>

**المثال العملي (Example):**

    git push origin v1.0
    git push origin v2.0

_(بمجرد تنفيذ الأمر، إذا ذهبت إلى صفحة مستودعك على `GitHub`، ستجد قسم جديد يسمى `Tags`، وبداخله الإصدارات التي قمت برفعها جاهزة للتحميل كملفات `ZIP`)._

---

> **💡 نقطة متقدمة (الفروع والإصدارات):**
> لو لاحظت في المشاريع الكبيرة (زي `Bootstrap`)، كل إصدار كبير بيكون ليه `Branch` (فرع) خاص بيه. يعني الإصدار الرابع ليه فرع `v4-dev`، والإصدار الخامس ليه فرع `main`. ده بيمنع التداخل (Confusion)، بحيث التحديثات اللي بتتعمل على الإصدار الجديد متأثرش على استقرار الإصدار القديم، وكل `Tag` بيكون مربوط بالفرع بتاعه بدقة.

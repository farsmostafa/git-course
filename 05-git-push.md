# 📝 رفع التعديلات للمستودع السحابي وفهم المصطلحات (الدرس الخامس)

في الدرس ده، بننتقل من مرحلة العمل المحلي (على جهازك) إلى مرحلة العمل السحابي، ونتعلم إزاي نرفع الشغل بتاعنا على `GitHub` عشان باقي الفريق يقدر يشوفه، وهنفهم كمان المصطلحات اللي ممكن تلخبطك في البداية.

## 1. مصطلحات الربط السحابي (Remote Terms)

قبل ما نرفع أي كود، لازم نفهم إحنا بنتعامل مع إيه بالظبط، لأن فيه مصطلحات بتظهر دايماً في الأوامر:

### أ. ما هو الـ `origin`؟

* **المفهوم:** هو مجرد "اسم مستعار" (Alias) أو اختصار (Nickname) للرابط الطويل جداً الخاص بمستودعك على `GitHub`.
* **السبب:** تخيل إنك كل ما تيجي ترفع كود تضطر تكتب الرابط ده `https://github.com/farsmostafa/git-course.git`، هيكون موضوع متعب جداً. فبرنامج `Git` بيسهل عليك وبيسمي الرابط ده `origin`. فكلمة `origin` معناها ببساطة: "المكان اللي هرفع عليه على النت".

### ب. إيه الفرق بين `main` و `master`؟

* **المفهوم:** الاتنين هما مجرد أسماء لـ "الفرع الأساسي" (Primary Branch) في مشروعك.
* **سبب الاختلاف:** زمان، كان الاسم الافتراضي لأي مشروع جديد في `Git` هو `master` (وده اللي بشمهندس أسامة شغال بيه في الفيديو). لكن من فترة، منصة `GitHub` وكل الشركات الكبيرة قرروا يغيروا الاسم الافتراضي ده وخلوه `main`.
* **تنبيه هام:** لو مشروعك مبني على فرع `main` (زي ما حصل معاك)، وحاولت ترفع بكتابة الأمر القديم، هيظهرلك خطأ بيقول `error: src refspec master does not match any`. وده معناه ببساطة إن `Git` بيقولك: "أنا دورت في مشروعك على فرع اسمه master ومولقيتوش!". الحل إنك دايماً ترفع على اسم الفرع الحقيقي بتاعك وهو `main`.

## 2. أوامر الاستكشاف قبل الرفع

دايماً اتأكد من مكانك واسم الفرع اللي إنت عليه قبل ما ترفع، عشان متعملش مشاكل في المشروع:

**1. لمعرفة الفرع الحالي:**

عشان تتأكد إنت واقف على فرع إيه، بتكتب الأمر ده، وهيظهرلك نجمة `*` جنب اسم الفرع الحالي.

**القاعدة الأساسية والمثال العملي:**

    git branch

**2. لمعرفة المستودع السحابي المتصل بجهازك:**

عشان تتأكد إن جهازك مربوط صح بالرابط بتاعك على `GitHub`.

**القاعدة الأساسية والمثال العملي:**

    git remote -v

## 3. الدورة الكاملة لرفع التعديلات (The Workflow)

عشان ترفع أي ملفات جديدة (زي ملفات النوتس بتاعتك) أو أي تعديل عملته، لازم تمشي على الـ 3 خطوات دول بالترتيب وبدون ما تفوت خطوة:

### الخطوة الأولى: الإضافة للمسرح (Add)

وهي نقل الملفات من مجلد العمل العادي إلى منطقة التجهيز (`Staging Area`).

**القاعدة الأساسية (Syntax):**

    git add <file_name>

**المثال العملي (Example):** (النقطة هنا تعني إضافة كل الملفات الجديدة والمعدلة في المجلد الحالي مرة واحدة)

    git add .

### الخطوة الثانية: الحفظ في المخزن المحلي (Commit)

وهي حفظ التعديلات دي بشكل رسمي كلقطة (`Snapshot`) في مستودعك المحلي، ولازم تكتب رسالة توضح إنت عملت إيه.

**القاعدة الأساسية (Syntax):**

    git commit -m "<message>"

**المثال العملي (Example):**

    git commit -m "docs: add git and github notes from lesson 1 to 4"

### الخطوة الثالثة: الدفع للسيرفر (Push)

وهي رفع كل الـ `Commits` اللي حفظتها محلياً إلى مستودعك على `GitHub`.

**القاعدة الأساسية (Syntax):**

    git push <remote_name> <branch_name>

**المثال العملي (Example):**

    git push origin main

## 4. ملاحظات هامة بعد الرفع

* **رسالة Everything up-to-date:** لو ظهرتلك الرسالة دي وقت الرفع، معناها إنك إما نسيت تعمل `commit` جديد للتعديلات بتاعتك، أو إن الكود اللي على جهازك هو هو نفسه اللي موجود على `GitHub` بالفعل ومفيش أي حاجة جديدة تترفع.
* **التسجيل الزمني (Time Tracking):** التاريخ اللي بيظهر جنب الـ `commit` على موقع `GitHub` هو تاريخ "وقت الحفظ على جهازك"، مش وقت الرفع. يعني لو عملت الكود وحفظته النهاردة، ورفعته على الموقع بعد أسبوع، هيظهر للناس إنه مكتوب من أسبوع، وده بيحفظ حقك ومجهودك بالثانية.
* **تتبع التعديلات بدقة:** لما تضغط على أي `commit` في الموقع، هيوريك إيه اللي اتغير في الملفات بالظبط (السطور اللي لونها أخضر وفيها علامة `+` معناها كود جديد انضاف، والسطور اللي لونها أحمر وفيها علامة `-` معناها كود اتمسح).
